### Landscape check

WebSearch: "Android exact alarm reminder habit app AlarmManager battery optimization
2026 reliability."
- **[Layer 1]** Tried-and-true: AlarmManager + foreground services are the standard
  Android mechanism for time-based reminders; this is not exotic.
- **[Layer 2]** Current discourse: since Android 12, `SCHEDULE_EXACT_ALARM` gates exact
  alarms; **since Android 14 it is denied by default** and must be explicitly granted
  by the user via Settings — a stricter rule than the design doc's Open Questions
  assumed. Doze mode can still defer alarms unless `setExactAndAllowWhileIdle` is used.
  Google's own guidance: prefer `WorkManager` over `AlarmManager` for anything that
  doesn't need wall-clock precision.
- **[Layer 3 — EUREKA]** The design doc reaches for *exact* alarms without asking
  whether exactness is actually required. A "have you read today" nudge firing at 9:03
  instead of 9:00 costs nothing. Dropping the exact-alarm requirement entirely
  (inexact `set()`/`setWindow()` or `WorkManager`) sidesteps the whole
  `SCHEDULE_EXACT_ALARM` permission surface — the single biggest reliability risk in
  the plan disappears if precision is never claimed in the first place. This was
  independently confirmed by the dual-voice subagent below — logged as a real eureka,
  not just a subagent opinion.

Sources: [Esper: Android 13 exact alarm restrictions](https://www.esper.io/blog/android-13-exact-alarm-api-restrictions) · [Android Developers: Schedule alarms](https://developer.android.com/develop/background-work/services/alarms) · [AlarmManager API reference](https://developer.android.com/reference/android/app/AlarmManager)

### Section 1: Architecture Review

```
SYSTEM ARCHITECTURE
                     ┌─────────────────┐
                     │   MainActivity   │  (Compose: Home / Settings / History)
                     └───────┬──────────┘
                             │ observes StateFlow / binds
                     ┌───────▼──────────┐        ┌────────────────────┐
                     │ ReadingTimerService│◄──────┤  AlarmScheduler     │
                     │ (foreground)      │       │ (single source of   │
                     └───────┬──────────┘        │ truth, called by ↓)│
                             │ read/write         └───┬─────────┬──────┘
                     ┌───────▼──────────┐   schedules │         │ schedules
                     │   Room (AppDb)    │◄────────────┘         │
                     │ Config/Daily/     │                       │
                     │ Session tables    │        ┌──────────────▼──┐
                     └───────▲──────────┘         │ BootReceiver     │
                             │ query completed?    │ RolloverReceiver│
                     ┌───────┴──────────┐          └──────────────────┘
                     │  NudgeReceiver    │◄── fires from AlarmManager
                     │ (posts/skips      │
                     │  notification)    │
                     └───────────────────┘
```

**Data flow (timer start), all four paths:**
- Happy: tap Start → ViewModel.start() → bind ReadingTimerService → service reads
  `DailyProgress` (or creates it from `ReadingConfig`) → coroutine tick begins →
  Room persisted every ~5s → StateFlow → UI countdown updates.
- **Nil path — GAP:** no `ReadingConfig` row exists at all (very first app launch,
  before Settings has ever been opened). Design doc doesn't specify default seeding.
  **Fix (promoted into scope):** seed a default `ReadingConfig` row (Sun–Thu, 15 min)
  at first-launch/app-init, not lazily inside the timer-start path.
- **Empty path:** no `DailyProgress` for today — already handled (service creates one
  from config). No gap.
- **Error path — GAP:** Room write throws (disk full/corruption) mid-tick. Not
  rescued anywhere in the design. **Fix:** wrap persistence in try/catch; on failure,
  keep the in-memory countdown running (don't crash the service) and surface a
  non-blocking "progress isn't saving right now" state — see Section 2.

**State machine (DailyProgress):** `NOT_STARTED → IN_PROGRESS → PAUSED ⇄ IN_PROGRESS
→ COMPLETED`. `IN_PROGRESS` is only held in the service's in-memory `isRunning` flag,
not persisted — if the process dies while running, the safe default on next launch
must be **PAUSED** (show remaining time, require an explicit tap to resume), never
silently resume as if still running. This is implied by the design doc but should be
a stated invariant, not an accident of implementation.

**Coupling:** ViewModels ↔ Service (bound service, justified), `AlarmScheduler` called
from 3 triggers (rollover/boot/settings-save) — correctly a single shared helper per
the design doc, not duplicated logic.

**Scaling:** irrelevant at this data volume (a few thousand rows over years) — noted,
not a real risk, evaluated rather than silently skipped per the anti-skip rule.

**Single point of failure:** the on-device Room DB file — no backup/export exists.
Low-severity given personal use, but real: a factory reset or lost phone destroys
the entire reading history the app exists to build. Promoted to TODOS.md, not core
scope.

**Security architecture:** no auth needed (on-device, single-user) — see Section 3.

**Production failure scenario (the review's #1 finding, confirmed by both voices):**
Android 14+ denies `SCHEDULE_EXACT_ALARM` by default; OEM battery management can kill
foreground services outright. If either happens, **nudges silently stop firing and
nothing tells the user.** For an app whose entire value proposition is reliability,
this is the single most important risk in the whole plan. **Resolved via the eureka
above: drop the exact-alarm requirement (inexact scheduling is sufficient for an
hourly nudge), and add a visible permission/battery-exemption status row so any
remaining failure mode is at least visible, not silent.**

**Rollback posture:** no server deploy; "rollback" = reinstalling an older APK. Room
schema changes MUST use real Migrations (see Section 9) or an update silently wipes
the user's history — this is the closest thing to a "deploy" risk this app has.

### Section 3: Security & Threat Model

Attack surface: none externally (no network, no server, on-device only). Local
threats only:
- **Exported components:** `BootReceiver` must be exported (system broadcast
  requirement) but scoped to `BOOT_COMPLETED` only. `NudgeReceiver` and
  `RolloverReceiver` should be `android:exported="false"` — they're only triggered via
  `AlarmManager` `PendingIntent`s, which doesn't require export. **Finding (Low):**
  design doc doesn't state this explicitly; add to the manifest plan so it isn't
  accidentally left exported (a real, if low-severity, local intent-injection surface
  if another app could fire your receivers).
- **Data classification:** reading-habit metadata only, no PII, no credentials,
  standard Android app-private storage sandboxing is sufficient. No dependency risk
  beyond standard Jetpack libraries (Room, WorkManager, Compose) — no injection
  vectors (Room's annotated DAOs, no raw SQL from user input anywhere in the design).

Overall: **Low** threat profile. One low-severity finding (exported-component
hygiene), promoted into scope since it's a one-line manifest fix.

Per-item test plan (representative, not exhaustive):
- **Timer resume** — unit test: given `remainingSeconds=340` persisted, tapping start
  resumes from 340, never resets to the full target. Failure test: Room write throws
  → service logs and keeps the in-memory countdown alive rather than crashing.
- **Streak calculation** — unit test with a fixture history including non-enabled
  days; a non-enabled day must never break the streak; today doesn't count until
  `completed=true`.
- **Nudge suppression** — Robolectric/instrumented test: `completed=true` → nothing
  posted; a session `IN_PROGRESS` at the scheduled hour → nothing posted (new case
  from Section 4).
- **Double-tap guard** — instrumented UI test: two rapid taps result in exactly one
  service start.
- **2am-Friday test:** a full day-cycle instrumented test — fast-forward through
  00:01 rollover → 9:00 nudge → start/stop/resume → completion → nudge suppression,
  using a fake clock / shadow AlarmManager (Robolectric), never real wall-clock waits.
- **Chaos test:** `adb shell am kill` mid-countdown; verify `remainingSeconds`
  persisted within the last checkpoint window and resumes correctly.

Test pyramid: mostly unit (streak calc, day-mask logic, repository), a handful of
Robolectric/instrumented tests for the service and receivers — no E2E device farm
needed for a personal app. Flakiness risk: any test that waits on a real AlarmManager
firing in wall-clock time — must use a fake clock/shadow instead.

### Section 8: Observability & Debuggability Review

Full production observability infra (dashboards, alerts, on-call runbooks) would be
over-engineering for a solo personal app — explicitly out of scope. What actually
matters: **when nudges silently stop firing, the user has no way to know why.** The
single highest-value observability investment here is a Settings-screen "Notification
status" row showing (a) exact-alarm/inexact scheduling active? (b) battery
optimization exemption granted? — turning an invisible failure into a visible,
fixable one. Promoted into core scope. Secondary: a consistent `Log.d/w` tag

### Section 1: Architecture Review

- **Component boundaries:** `ReadingTimerService` (timer authority) ↔ `AlarmScheduler`
  (nudge scheduling authority) ↔ Room (`ReadingRepository` as the single access point
  for both) is a clean split — no component owns another's responsibility. Sound.
- **[P1] (confidence: 9/10)** design doc lines 117-119 (Nudge scheduling) — the plan as
  written uses `AlarmManager.setExactAndAllowWhileIdle`, which requires
  `SCHEDULE_EXACT_ALARM`, denied by default on Android 14+ (confirmed by Phase 1's
  landscape search). **Fix (already adopted as binding from Phase 1 premise gate):**
  replace with `AlarmManager.setWindow()` (inexact, ~a few minutes of slop, which is
  fine for an hourly nudge) or migrate the whole nudge-firing mechanism to a
  `WorkManager` `PeriodicWorkRequest`-style approach using expedited one-time work
  requests scheduled by the daily rollover job. **Recommendation: `setWindow()`**, not
  WorkManager, for this specific case — WorkManager's minimum periodic interval is 15
  minutes and its exact fire time isn't guaranteed either way, so it buys nothing over
  `setWindow()` here while adding a second scheduling system to reason about. Simpler
  wins (engineering preference: explicit over clever).
- **[P1] (confidence: 8/10)** design doc line 99 (`ReadingTimerService`) — Android 14
  requires foreground services to declare a specific `foregroundServiceType` in the
  manifest (`dataSync`, `mediaPlayback`, etc.) and request the matching runtime
  permission for some types. A reading timer doesn't cleanly map to any predefined
  type — **`specialUse`** is the correct fit (requires a manifest `<property>` declaring
  the use case string, e.g. "user-initiated reading timer"), but this was entirely
  unaddressed in the design doc. Missing this means the app crashes with a
  `MissingForegroundServiceTypeException` on API 34+ the moment `startForeground()` is
  called. **Critical, not previously flagged anywhere.**
- **[P1] (confidence: 9/10)** design doc — `POST_NOTIFICATIONS` runtime permission
  (Phase 2 finding, carried forward): if denied or never requested, `NotificationManager.
  notify()` silently no-ops on API 33+. **Fix:** request at first launch (not buried in
  Settings) with a clear one-line explanation ("so I can nudge you to read"), and if
  denied, the Home screen must show a visible "Notifications are off — nudges won't
  fire" banner with a deep link to app settings. A silent failure here defeats the
  entire product.
- **[P2] (confidence: 7/10)** — one realistic production failure scenario per new
  codepath:
  - `BOOT_COMPLETED` receiver: if it throws (e.g. a Room migration exception on first
    access after an OS update), the whole day's nudges silently never get scheduled,
    with zero user-visible signal. **Fix:** wrap the boot-time scheduling call in a
    try/catch that, on failure, still lets the app function normally when opened
    manually — do not let a boot-time crash brick reads for the day. Log to
    `Log.e` for `adb logcat` debugging (matches the plan's existing no-crash-reporting
    stance).
  - Daily rollover alarm: if the device is off/dead at 00:01 and turns back on at 08:00,
    today's `DailyProgress` row and nudge alarms never got created. **Fix:** the
    `BOOT_COMPLETED` receiver and the app's own `onCreate`/first-open-of-the-day path
    should both independently check "does today have a `DailyProgress` row and
    scheduled alarms?" and self-heal if not — don't rely solely on the midnight alarm
    firing.
  - OEM battery killers (Samsung/Xiaomi/Huawei aggressive process killing): **no code
    fix exists.** This is the plan's one un-rescuable gap, already correctly identified
    in Phase 1. Confirmed here at the architecture level: the best available mitigation
    is a first-launch "Disable battery optimization for this app" prompt (already in
    the design doc's Next Steps as item 7) plus the Section-8 status row (Phase 1
    cherry-pick). No further code-level rescue is possible — this is a platform
    limitation, not a design flaw.
- **Security architecture:** no auth, no network, no PII beyond "did I read on this
  date" — appropriately minimal. Room database is app-private storage (standard Android
  sandboxing); no additional encryption needed for this threat model (personal device,
  personal habit data, not health/financial data). No issues.
- **Distribution architecture:** already resolved (sideload, no CI/CD needed) — correct
  scope, not a gap.
