## Problem Statement

You want to read a book for 15 minutes every Sunday–Thursday, but you forget. You want
an app that pokes you hourly (9:00–13:00) on your reading days until you've done today's
15 minutes, with a timer you can pause and resume across the day.

## What Makes This Cool

The nudge disappears the moment you've actually read — no guilt-notifications after
you've already done the thing. And the timer remembers exactly where you left off, so
"5 minutes on the bus + 10 minutes at lunch" counts the same as one sitting.

Copy tone throughout (nudges, completion message): **invitational, not urgent** — e.g.
"15 minutes today?" / "Nice — today's reading is done." A missed day is never framed as
a broken streak; total days completed and current streak are tracked and shown as
separate numbers, so one bad week doesn't erase months of history from view.

## Constraints

- Personal, single-user app. No accounts, no server — everything lives on your phone.
- Android only, for now (native Kotlin project already scaffolded in this repo).
- Nudges must fire at 9:00, 10:00, 11:00, 12:00, 13:00 on configured days, **only if**
  today's reading isn't done yet — and must keep firing even if the app is closed or
  the phone was rebooted.
- Configurable: which days count (default Sun–Thu) and target duration (default 15 min).
- Timer: single start/stop toggle button. Starting begins counting down from the
  configured duration (or from wherever it was paused). Stopping pauses it in place.
  Reaching zero marks today as done and silences the rest of today's nudges.
- You'll sideload it yourself (Android Studio / `adb install`) — no Play Store listing,
  no signing pipeline needed.

## Premises

1. Local-only data, no backend — **agreed**.
2. Real OS-scheduled nudges (AlarmManager), not an in-app-only timer — **agreed**.
3. A nudge is just a reminder; tapping it opens the app, it doesn't force anything — **agreed**.
4. Personal sideload distribution, no Play Store — **agreed**.

## Approaches Considered

### Approach A: Minimal viable
Kotlin + Compose, single Activity, DataStore for state, AlarmManager for nudges,
foreground service for the running timer. Fastest path to a working nudge loop, but
no history — only tracks "did I read today."

### Approach B: Ideal architecture (chosen)
Same scheduling/timer core as A, plus a Room database that keeps a full history of
reading sessions and days completed, and a stats/streak screen. More to build up
front, but the habit-tracking payoff (streaks, history) is worth it for a habit app —
"did I read today" without "how's my streak going" is only half the product.

### Approach C: Notification actions (not chosen)
Same core, but the nudge notification itself has a "Start timer" button so you never
have to open the app to start reading. Highest delight-per-tap, but fiddlier
(notification-action → foreground-service handoff has real edge cases across OEMs).
Worth revisiting later as a v2 enhancement once the core loop is solid — see Open
Questions.

## Recommended Approach

**Approach B.** It reuses every mechanism from A (AlarmManager, foreground service,
DataStore-equivalent) but stores state in Room instead of flat prefs, which costs
almost nothing extra up front and unlocks streaks/history for free. The stats screen
can be v1.1 if you want to ship the core loop first — the schema decision is what
matters now, and Room vs. DataStore is a one-time choice worth getting right early.

## Architecture

### Data model (Room)

```
ReadingConfig (singleton row, id = 0)
  - enabledDaysMask: Int        // bitmask, Sun=1, Mon=2, ... Sat=64; default 0b0011111 (Sun–Thu)
  - targetSeconds: Int          // default 900 (15 min)

DailyProgress (PK = date, ISO "yyyy-MM-dd", device-default timezone / LocalDate.now())
  - date: String
  - targetSeconds: Int          // snapshot of config's targetSeconds when this day was created
  - remainingSeconds: Int       // updated once at Stop and once at Completion (see Timer mechanics)
  - completed: Boolean
  - completedAt: Long?          // epoch millis
  - activeSessionStartedAt: Long?  // set when a session is running, cleared on Stop/Completion;
                                    // a non-null value with no matching close on next app launch
                                    // means the process died mid-session — reconcile, don't lose it

ReadingSession (PK = autoincrement id)
  - date: String                // @ForeignKey -> DailyProgress.date (CASCADE), @Index("date")
  - startedAt: Long
  - endedAt: Long
  - secondsAdded: Int

Stats (singleton row, id = 0)
  - totalCompletedDays: Int     // lifetime count, updated transactionally when a day completes
  - currentStreak: Int          // consecutive completed days walking back from today,
                                 // skipping non-enabled days; reset on a missed enabled day
```

`totalCompletedDays` and `currentStreak` are stored as separate values, not folded into
one "streak" concept — a missed day resets `currentStreak` but never decreases
`totalCompletedDays`, so historical progress is never erased from view (see the
invitational-tone note above). `ReadingSession` gives the "3 short sessions today"
level of detail for the stats screen, without being load-bearing for anything else.

**Migration policy:** real `Migration` objects from v1 onward — never
`fallbackToDestructiveMigration()`. The entire point of this schema is long-term
history; silently wiping it on a future schema change would defeat the app's purpose.

### Timer mechanics

- `ReadingTimerService` (foreground service, `foregroundServiceType="specialUse"` with
  the required manifest justification string — Android 14 requires a declared type or
  `startForeground()` throws) owns a single source of truth: `remainingSeconds:
  StateFlow<Int>` + `isRunning: StateFlow<Boolean>`. `START_NOT_STICKY` — no OS
  auto-restart; the reconciliation path below handles recovery instead.
- **Timer model: timestamp-delta, not tick-decrement.** On Start, record
  `sessionStartedAt = now()` (persisted to `DailyProgress.activeSessionStartedAt`
  immediately — durable before a single second ticks). The displayed countdown is
  computed reactively as `remainingSeconds - (now() - sessionStartedAt)`, driven by a
  1Hz UI tick for display only — never the source of truth and immune to
  coroutine/Doze scheduling drift. No periodic Room writes while running.
- **Start:** if no `DailyProgress` row exists for today, create one from current
  `ReadingConfig.targetSeconds`. Launch the foreground service (ongoing notification
  shows live countdown, its own low-importance silent notification channel — see
  Nudge scheduling), set `activeSessionStartedAt = now()`.
- **Stop:** compute actual elapsed as `now() - activeSessionStartedAt`, write the new
  `remainingSeconds` once, clear `activeSessionStartedAt`, write a `ReadingSession` row
  (`startedAt`/`endedAt`/`secondsAdded`), stop the foreground service. Exactly one Room
  write for the whole segment — no periodic-write data-loss window to reason about.
- **Resume:** pressing start again re-reads `remainingSeconds` from Room and continues
  the countdown from there via a fresh `sessionStartedAt` — never resets to the full
  target.
- **Completion:** when computed remaining hits 0, mark `DailyProgress.completed = true`,
  `completedAt = now`, clear `activeSessionStartedAt`, update `Stats.totalCompletedDays`
  and `Stats.currentStreak` transactionally, stop the service, cancel any of today's
  still-pending nudge alarms, and post a short "Nice — today's reading is done"
  confirmation notification (invitational tone, own notification channel).
- **Crash/kill recovery:** if the process dies mid-session, `activeSessionStartedAt` is
  left non-null with no matching Stop. On next app launch, detect this and reconcile:
  credit elapsed time up to the crash, close out the `ReadingSession` row, clear the
  marker — never silently lose a session or leave the UI in an ambiguous state.

### Nudge scheduling

- **No exact alarms.** A nudge firing a few minutes late costs nothing, and exact
  alarms (`SCHEDULE_EXACT_ALARM`) are denied by default on Android 14+ — claiming
  exactness would mean fighting a permission the feature doesn't actually need.
  Scheduling uses `AlarmManager.setWindow()` (inexact) throughout — for both the daily
  rollover job and the 5 per-day nudge alarms. Expect actual delivery within OS
  scheduling tolerance (can be 30+ min under Doze on an idle device), not to-the-minute.
- A daily rollover job (fires ~00:01, inexact window) runs every night: if tomorrow is
  an enabled day, it creates tomorrow's `DailyProgress` row and schedules that day's
  nudge alarms (9/10/11/12/13:00).
- A `BOOT_COMPLETED` receiver re-runs the same scheduling logic after a reboot (alarms
  don't survive reboots on their own), wrapped in try/catch — a boot-time exception
  must never silently brick the day's nudges; the app still schedules correctly on next
  manual open regardless. Both boot and app-open independently self-heal: "does today
  have a row + scheduled alarms? If not, create them" — not solely reliant on the
  midnight job having fired.
- Each nudge alarm fires a `BroadcastReceiver` that checks `DailyProgress.completed`
  for today before posting anything — if already done, it's a no-op. This is what
  makes "stop nagging once I've read" work without canceling/re-scheduling constantly.
- **Notification IDs:** the 5 hourly nudges reuse one fixed notification ID per day
  (each new nudge replaces the previous one in the shade, they don't stack). The
  ongoing-timer notification and the completion notification each use their own
  distinct, fixed IDs. Three separate notification channels: nudges (default
  importance, sound), ongoing timer (low importance, silent), completion (default
  importance) — so you can mute the persistent timer notification without muting
  nudges.
- Editing Settings (days or duration) immediately re-runs the scheduling logic for
  today and going forward — a duration change never retroactively changes a day whose
  `DailyProgress` row already exists (already-snapshotted `targetSeconds` stands); it
  only affects days without a row yet.

### Screens

Visual style: **Material You / dynamic color** — system-derived palette, no custom
brand to design or maintain, right-sized for a personal utility app.

1. **Home** — four states: not configured yet (first launch), enabled day not yet done
   (remaining time + timer), enabled day done (invitational confirmation), and
   **non-enabled day** (today isn't in your configured schedule — no timer/nudges, but
   you can still manually start a session if you want to read anyway; doing so creates
   today's `DailyProgress` row same as any other day). If `POST_NOTIFICATIONS` was
   denied, a persistent banner ("Notifications are off — nudges won't fire") with a
   deep link to system notification settings overlays whichever of the 4 states applies.
2. **Settings** — day-of-week picker (default Sun–Thu) and duration picker (default
   15 min, validated > 0).
3. **History/Stats** — `Stats.currentStreak` and `Stats.totalCompletedDays` shown as
   two distinct numbers (never one merged "streak"), calendar-style view of completed
   days, list of individual sessions per day.

## Open Questions

- Some OEMs (Samsung, Xiaomi, etc.) aggressively kill foreground services/background
  alarms unless the app is battery-exempted — a first-launch "disable battery
  optimization for this app" prompt (Next Steps #7) is necessary, not optional, given
  this app's whole point is reminding you of things you'd otherwise forget. This is a
  platform limitation with no full code fix; the exemption prompt plus a visible
  Settings status row are the only available mitigation.
- If the target phone is Xiaomi/MIUI specifically: MIUI has a separate "Autostart"
  permission distinct from battery-optimization exemption, and can block
  `BOOT_COMPLETED` without it. Add a note/deep-link if applicable; skip otherwise.
- Approach C's notification "Start timer" action stays a deferred v2 (see TODOS.md) —
  reaffirmed after CEO review raised reprioritizing it ahead of History/Stats; the
  streak/history payoff is why Approach B was chosen over Approach A in the first
  place, so History/Stats stays first.

## Success Criteria

- On an enabled day, if you haven't read yet, you get a notification at each of
  9/10/11/12/13:00 — and none after you've completed the day.
- You can start, stop mid-session, and resume later the same day without losing
  progress, and completing the target marks the day done and stops further nudges.
- Config changes (days, duration) take effect for future scheduling without needing
  to reinstall or restart the app.
- Reading history/streak is visible and survives app restarts and phone reboots.

## Distribution Plan

Personal sideload: build a debug/release APK via Android Studio or `./gradlew
assembleDebug` and `adb install` it onto your own phone. No Play Store listing,
no signing pipeline, no backend to stand up.

## Next Steps

1. Room schema (`ReadingConfig`, `DailyProgress`, `ReadingSession`, `Stats`) + DAOs +
   real `Migration` objects (never destructive fallback) + `@ForeignKey`/`@Index` on
   `ReadingSession`.
2. `ReadingTimerService` (foreground service, `specialUse` type, `START_NOT_STICKY`)
   with timestamp-delta start/stop/resume, `activeSessionStartedAt` crash recovery, and
   `POST_NOTIFICATIONS` permission request (first launch, with a denied-state banner).
3. AlarmManager scheduling (`setWindow()`, daily rollover job + boot receiver with
   try/catch + self-heal check + per-alarm "is today done?" check), 3 notification
   channels, fixed per-day nudge notification ID.
4. Home screen (Compose): 4 states (not configured / not done / done / non-enabled day)
   + notifications-off banner + timer + toggle button.
5. Settings screen: day picker + duration picker (validated > 0), wired to
   `ReadingConfig` and re-triggering today's scheduling on save.
6. History/Stats screen: `currentStreak` + `totalCompletedDays` (separate) + session
   list.
7. Battery-optimization exemption prompt (first-launch, one-time); Xiaomi/MIUI
   Autostart note if applicable to the target device.
